# heartrate-sensor-parser 技能使用手册 V2.4.5

## 一、技能概述
专用 XOSS 心率设备 BLE 调试日志离线解析工具。自动识别 2/4/6/8 字节（及扩展长度）全部心率报文，提取设备固件/SN/电量等参数，批量计算真实瞬时心率、RR 间期、全量 HRV 时域指标（SDNN/RMSSD/pNN50/pNN20/SDSD/SDARR/CVRR/HRV三角指数/Tin），**自动识别场景（睡眠/运动）并据此动态适配分析策略和报告内容**，动态划分心率区间（静息/热身/有氧/高强度/极限），检测心律异常（RR 间期突变/疑似早搏），**对运动场景额外做运动模式细分识别（骑行/跑步/游泳/爬山/混合）**，输出标准化 Excel 三表 + CSV + JSON + 分段运动心律波动分析报告 + HTML 可视化报告（**Chart.js 已内联，离线/沙箱预览可直接出图，无需联网**）。

### V2.4.5 核心变更（对比 V2.4.4）

**`_user_meta.json` 查找路径扩展 + 仓库根用户偏好共享**。V2.4.4 只在「日志所在目录 / 输出目录 / 输出目录父级」查 `_user_meta.json`，跨工作区（日志放在 A 目录、skill 装在 B 目录）时读不到用户偏好；本版本把查找路径延伸到脚本仓库根及其父级，方便"写一次 HRmax，全设备共享"。

1. **查找顺序（首个命中生效）** —— `process_single_log()` 在未显式传入 `hr_max_override`（CLI `--hr-max`）时按以下顺序查找 `_user_meta.json`：
   1. 日志所在目录及其父级
   2. 输出目录及其父级
   3. **脚本仓库根（`Path(__file__).resolve().parent` = `scripts/`，取其 `.parent` 即 skill 仓库根）及父级**（V2.4.5 新增）
   路径去重后按序尝试 `json.loads`，取 `dict.hr_max` 字段作为 `hr_max_override`。

2. **推荐用法（个人偏好设置）** —— 在 skill 仓库根（`/path/to/heartrate-sensor-parser/_user_meta.json`）里写一次：
   ```json
   {
     "name": "heartrate-sensor-parser",
     "hr_max": 180
   }
   ```
   之后所有工作区、所有日志都会自动按 `hr_max=180` 缩放运动负荷区间，`_meta.hr_max_source = "override"`。老年人 / 低 HRmax 用户特别推荐这种一次性配置。

3. **优先级不变** —— CLI `--hr-max` 仍最高优先级，覆盖 `_user_meta.json`；两者都缺失时走 V2.4.4 的 `max(HR_MAX_DEFAULT=190, hr_p95)` 策略。

4. **回归影响**：分类逻辑、门控、置信度均不变；仅新增一条"仓库根 user_meta 自动读取"路径。已验证 `0701-pb` 未带 `--hr-max` 参数时能自动读到仓库根 `_user_meta.json` 的 `hr_max=180`，`hr_max_source: override` 生效。

### V2.4.4 核心变更（对比 V2.4.3）

**HRmax 参考值三级策略：显式覆盖 → 190 兜底 → P95 抬高**。修复"日志未达本人 HRmax 时 P95 缩放出的极限档门槛过低（149bpm），稳态跑步被算成 40~55% 极限占比"的失真。

1. **HRmax 参考值选取顺序** —— `split_exercise_segment(rr_rows, hr_max_override=None)` 按以下顺序确定 `hr_max_ref`：
   1. `hr_max_override`（≥100 bpm）：CLI `--hr-max` 或日志/输出目录 `_user_meta.json.hr_max` 显式指定，最准。
   2. 未指定 → `max(HR_MAX_DEFAULT=190, hr_p95)`：默认按成年人 HRmax 常规上限 190 兜底，仅当本次日志真出现过 ≥190 bpm 的心率片段才用 P95 抬高。
   3. 极端兜底：所有心率序列均 <100（纯静息数据）时仍走 190，不采用 P95。
   `_meta` 新增 `hr_max_source` 字段：`override / p95 / default_190 / default_190_no_signal`，便于报告端追溯选源。

2. **CLI 与元数据入口** —— `parse_heart_rate_log.py` 新增 `--hr-max` 参数（float）；`process_single_log` 增 `hr_max_override` 形参。若未指定，按「日志所在目录及父级 / 输出目录及父级 / 脚本仓库根及父级」顺序查找 `_user_meta.json`，取 `hr_max` 字段。仓库根命中让 skill 用户可在 `_user_meta.json` 里写一次 `hr_max` 供所有工作区共享。

3. **默认值选择说明（"为什么是 190"）** ——
   - Tanaka 公式（`208 − 0.7·age`）在 30 岁 ≈ 187、25 岁 ≈ 190.5、20 岁 ≈ 194；成年运动人群普遍 185~195 之间。
   - 用户当前样本 hr_p95 集中在 157~167，若走"P95 直接当 HRmax"，极限门槛（0.9×p95）只有 141~151，稳态跑步/骑行心率轻松越过，导致极限档虚高 40~65%。
   - 190 作为兜底能让"未跑到极限"的日常样本落入合理档位分布（极限占比 <5%），同时对"真正跑到 ≥190"的极端样本仍会用 P95 抬高，两头兼容。
   - 老年人/低 HRmax 人群建议显式传 `--hr-max` 或写入 `_user_meta.json`，本轮不做年龄兜底。

4. **回归效果**（20 个样本，含 13 已标注）：
   - `0701-pb` 极限档 44.22% → 1.35%，高强度 15.99% → 47.67%；跑步/爬山/骑行的分类判定与置信度均不变。
   - 全部样本 `hr_max_ref = 190`（无一样本 P95 超过 190），档位分布回到"稳态跑步高强度 40~65% + 极限 <2%"的合理形态。
   - 分类结果表（跑步 4/4、爬山 4/4、骑行诚实降级、`0709-qx-1` 死结）与 V2.4.3 一致。

### V2.4.3 核心变更（对比 V2.4.2）

**跑步识别再强化：动态心率区间 + `sustained_high` 多维门控**。回应"高强度间歇跑被 drift 单打独斗压不住骑行/爬山"的死角，同时把所有阈值继续贴齐"个体化 + 大众可迁移"的目标。

1. **动态心率区间（方案 C，`parse_heart_rate_log.py`）** —— `split_exercise_segment` 从硬阈值 90/120/150/180 bpm 改为按 `hr_p95` × ACSM 比例（45/60/75/90%）动态划分。保留原区间 key 字符串（`高强度(150-180)` 等）以维持报告兼容，实际 bpm 边界由每个 zone 的 `阈值下界(bpm)/阈值上界(bpm)` 字段透出；同时挂 `seg_info["_meta"] = {hr_max_ref, hr_p95}` 便于回溯，两处遍历 `seg_detail` 处均已加 `if zone.startswith("_"): continue` 过滤。
   - **大众适用性**：ACSM 比例是学界通用的 HRmax 相对强度分级，跨年龄/性别/训练水平通用；用 `hr_p95` 而非固定 HRmax 估算，消除个体差异。

2. **`high_pct` 口径调整** —— 从"仅高强度 zone"改为"高强度 + 极限 两档合计（≥75% HRmax）"。跑步/爬山冲入极限档不能被漏计，否则 sustained_pct 恒零。

3. **跑步专属信号 `sustained_high` + 多维门控** —— 捕捉"持续中高强度 + 高变异 + 正/近零漂移 + dRR 尾部平滑"这一跑步指纹。四个乘子任一不满足即归零：
   - `sustained_pct = clip((high_pct - 25) / 30)`：25% 起效、55% 饱和。跑步实测 43~57%，骑行 ≤27.7%（3/3），中间干净分界带。
   - `var_running = clip((variability - 4) / 4)`：稳态骑行 30s HR std 2.5~3.6，跑步稳态段 5~13，作为跑步的必要变异门槛。
   - `drift_gate = clip((drift_pct + 0.20) / 0.10)`：drift_pct ≥ -0.10 完全放行、-0.20 完全拒绝；实测跑步收尾负漂 -0.05 通过，爬山下山段 -0.14~-0.23 大幅衰减，挡住 `0711_ps-2` 假阳性。
   - `tail_gate = 1 - clip((drr_tail_ratio - 30) / 30)`：跑步 P95(|dRR|)/median(|dRR|) 多在 3~40，骑行/间歇型样本因大量小 dRR 叠加偶发大跳变尾部飙到 50~120；`tail_gate` 是骑行 `0709-qx-2` (tail=60.7) 与跑步 `0702-pb` (tail=3.5) 唯一干净分离维度。**tail_ratio 是分布形态比率，天然不依赖 HRmax，跨人群普适**。
   - 跑步得分公式追加 `+ 1.5 * sustained_high`。

4. **大众适用性说明** —— 本次所有新增阈值均是"形态比率"或"占 HRmax 比例"，未引入任何 bpm 绝对硬值：
   - `sustained_pct` 起效 25%、饱和 55%：ACSM 强度分级下"持续 ≥75% HRmax"是跑步 vs 骑行的稳定分界，与个体最大心率无关。
   - `var_running` 起效 4bpm、饱和 8bpm：单位是 30s 滚动窗内的 HR 标准差，反映"步频颠簸"这一机械特征；就算是老年人低 HR 跑步，跑姿颠簸导致的 4-8bpm 窗内波动仍普适。
   - `drift_gate` 用 drift_pct（占 HRmax 比例）：`-0.10 ~ -0.20` 对应"每小时下降 10~20% HRmax"，跨人群一致。
   - `tail_gate` 阈值 30~60：是分布形态比率，非 bpm 绝对值，与个体基线心率解耦。

5. **回归验证**（13 个已标注样本 + 7 个未标注样本）：
   - ✅ 4/4 跑步：`0712-pb-1` 跑步 0.90 / `0712-pb-2` 跑步 0.72 / `0702-pb` 跑步 0.58 / `0701-pb` 跑步 0.58 IA
   - ✅ 4/4 爬山：`0711_ps-1` 爬山 0.84 / `0711_ps-2` 爬山 0.46 / `0712_ps-1` 爬山 0.83 / `0712_ps-2` 爬山 0.71（0711_ps-2 从"误判跑步 0.44"救回）
   - ⚠️ 骑行诚实降级：`0701-qx` / `0702-qx-1` / `0702-qx-2` / `0709-qx-2` 均判混合 LC（0709-qx-2 从"误判跑步 0.66 IA"改进为"混合 LC"）
   - ❌ 死结：`0709-qx-1` 判跑步 0.74（drift_pct=+0.46 + high_pct=65 + tail=13 + var=3.8，落在跑步特征区但 var<4 未真正触发 sustained_high；drift 单项就已压倒骑行，属纯 HR 无解，需 GPS/踏频）

### V2.4.2 核心变更（对比 V2.4.1）

**运动模式分类器重构：个体化归一化 + 爬山识别 + 短样本门控**。回应"drift 硬阈值只适合本人样本"和"爬山被误判为跑步/骑行"两个真实场景问题。

1. **个体化归一化 `drift_pct = drift / hr_p95`** —— 新增 `hr_p95`（1Hz HR 序列 P95，观察到的 HRmax 代理，避单点噪声）和 `drift_pct`（单位 1/h，物理含义"每小时心率爬升占本人 HRmax 的比例"）两个特征。所有涉及 drift 的阈值改用 drift_pct 表达，天然消除 HRmax 差异对判定的影响，对不同体能人群普适。
   - `drift_n`：`clip((drift+30)/60)` → `clip((drift_pct+0.15)/0.70)`
   - 语义连续：drift_pct ∈ [-0.15, 0.55] 为跑步甜蜜区，>0.55 归爬山领地。

2. **爬山专属信号 `uphill_extreme` + `downhill_signal`** —— 补齐爬山识别能力，将 ps 系列 4 个爬山样本从"错判跑步/骑行"修正为"正确识别爬山"。
   - `uphill_extreme = clip((drift_pct - 0.55) / 0.25)` × 5.0：捕捉上山特有的极端心率爬升速率（跑步渐进爬升 ≤0.52，上山 0.80~1.05）。
   - `downhill_signal = clip((-drift_pct - 0.05) / 0.15) × bim_n` × 3.0：捕捉上下山 session 特有的"负漂移 + 高双峰"组合（平地骑行负漂 bim<0.4 时几乎为零）。

3. **权重再平衡** —— 打破原公式对"骑行/爬山"的结构分偏袒，让跑步的 drift 信号真正起作用。
   - 骑行、爬山的 `struct_n` 权重：1.5 → 1.0（与跑步平权，间歇结构不再视为骑行专有）
   - 跑步的 `drift_n` 权重：1.0 → 2.5（drift 成为跑步主打信号）

4. **短样本时长门控 `duration_min < 5`** —— 极短样本 drift/climb/间歇等时域特征均不稳定，直接强制 low_confidence + poor_signal + dominant=混合，避免误判。修复 0707（1.6min 样本）在爬山信号引入后被爆表 drift（-497 bpm/h）错判为"爬山 0.65"。

5. **回归验证**（9 个已标注样本）：
   - 修正样本：0712-pb-1/pb-2（跑步 0.79/0.63）、0711_ps-1/ps-2 + 0712_ps-1/ps-2（爬山 0.49~0.89）全部与标注吻合。
   - 保留样本：0701、0709-2 稳定诚实降级为混合 LC+IA；0629 稳定混合。
   - 死结样本：0709-1（骑行标注但 drift+load 全部落在跑步区）判为跑步 0.74，属纯心率无解的特征空间死角，需融合 GPS/踏频。

   - 仍保留对**真实中途掉线**的捕捉：中途长静默同样会拉大（首末跨度 − 累加RR）。

### V2.3.0 新增内容
**场景自动识别 + 睡眠/运动双模式分析引擎**：`detect_scenario()` 四维评分（静息占比/平均心率/记录时长/夜间时段）判定睡眠 or 运动；睡眠场景切换专用 HRV 解读、睡眠结构、信号质量阈值与心内科分析。

1. **场景自动检测** — `CardioAnalyzer.detect_scenario()` 基于四维评分自动判定场景：
   - 静息占比 > 95%（+3分）
   - 平均心率 < 65 bpm（+2分）
   - 记录时长 >= 4小时（+2分）
   - 记录时段为夜间 22:00-10:00（+2分）
   - 总分 >= 5 → 睡眠场景，否则 → 运动场景

2. **睡眠场景专用分析方法**：
   - `interpret_hrv_metrics_sleep()` — HRV指标以睡眠生理参考范围解读（SDNN 60-120ms、RMSSD 30-80ms、pNN50 20-60%）
   - `analyze_sleep_structure()` — 分析NREM/REM期心率特征、心率尖峰簇集性、睡眠连续性评估
   - `analyze_anomalies_sleep()` — 区分体位改变/觉醒引起的V型心率响应与病理性心律失常
   - `generate_conclusions_sleep()` — 包含心脏自主神经评价、睡眠质量间接评估、OSA筛查提示
   - `assess_signal_quality_sleep()` — 睡眠场景专用的信号质量评估阈值

3. **报告模板动态适配**：
   - 标题自动切换：`心率分析报告` ↔ `睡眠心率 HRV 分析报告`
   - 摘要框切换：`运动负荷评估` ↔ `睡眠心率总体评估`
   - 图表标题切换：`运动负荷分布` ↔ `睡眠心率区间分布`
   - 心内科分析五大部分全部根据场景动态生成

```bash
python scripts/parse_heart_rate_log.py --log heart_rate_0701.txt
```

### 3. 完整参数命令（解析）

### 4. 批量解析文件夹


参数说明：
- `parse_heart_rate_log.py --log`：必填，心率 txt/log 日志路径
- `parse_heart_rate_log.py --out`：输出目录，默认 ./output

- `parse_heart_rate_log.py --batch`：批量日志文件夹路径
- `generate_report.py --json`：分析结果 JSON 路径（必填）
- `generate_report.py --csv`：心跳明细 CSV 路径（必填）
- `generate_report.py --out`：输出 HTML 路径（默认与 JSON 同目录）

## 四、支持报文规格
- 2 字节：Flags + HR + 0组RR
- 4 字节：Flags + HR + 1组RR
- 6 字节：Flags + HR + 2组RR
- 8 字节：Flags + HR + 3组RR
- 扩展长度（>8 字节）：自动识别多组RR

换算公式：
- RR_raw(小端16位) → RR_ms = RR_raw / 1024 * 1000
- 瞬时心率 = 60 / (RR_raw / 1024)

## 五、HRV 指标说明

| 指标 | 说明 |
|------|------|
| 摘要 | 交感神经只会持续抬高心率，无交替起伏；迷走神经会制造快速、小幅的心跳起伏，产生相邻 RR 长短差 |
| SDNN | 全部 RR 间期的标准差，反映交感+迷走神经总体心率变异性 |
| RMSSD | 相邻 RR 差值平方均值的根号，反映迷走神经短时调节功能，评估副交感最敏感指标 |
| SDSD | 相邻 RR 差值的标准差，反映迷走神经活性 |
| SDARR | 相邻 RR 差值绝对值的标准差，反映逐跳心跳快慢变化幅度的稳定程度 |
| pNN50 | 相邻 RR 差值 >50ms 的百分比 |
| pNN20 | 相邻 RR 差值 >20ms 的百分比 |
| CVRR | SDNN/平均 RR×100%，变异系数 |
| HRV 三角指数 | 总心跳数/最大频数 bin 的值，基于直方图频数分布 |
| Tin | RR 间期中位数，剔除早搏/异常干扰后的真实基础窦性心跳间隔 |

## 六、运动模式细分识别（classify_exercise_mode.py）

> 设计定位：**启发式估算，非硬判定**。单凭逐拍 RR/瞬时心率无法高置信区分运动模式（各模式 HR 区间高度重叠），故输出带置信度、低置信兜底（混合）与说明文案。睡眠/静息场景不调用本分类器（`applicable=False`）。

### 6.1 判别特征（9 个，标准库时域特征）
全部由 rr_ms / inst_hr 序列及汇总字段计算：

| # | 特征 | 计算 | 指向模式 |
|---|------|------|----------|
| F1 | intermittency 间歇指数 | 1Hz HR 序列中短窗(≤12s) HR 下跌 >15bpm 事件数 / 10min | 爬山/间歇 |
| F2 | spike_rate 尖峰率 | 短窗(≤12s) HR 上涨 >15bpm 事件数 / 10min | 爬山/间歇 |
| F3 | variability 平稳度 | 30s 滚动窗 HR 标准差均值(bpm)，越低越稳态 | 骑行 |
| F4 | bimodality 双峰性 | 2-means 方差缩减比(0~1)，越高越双峰 | 爬山 |
| F5 | mean_load 平均负荷 | 平均瞬时心率(bpm) | 游泳（偏低且平稳） |
| F6 | drift HR 漂移 | **分段斜率**：1Hz 序列分 5 段取均值再线性回归(bpm/h)，抗秒级噪声、反映整体趋势；原整段斜率保留为 `drift_raw` 供调试 | 跑步（平稳上行趋势） |
| F7 | 区间分布 | 透传 exercise_segments 各区间占比 | 多模式 |
| F8 | gap_ratio 连续性 | (有效RR首末跨度 − RR累加时长)/首末跨度，粗略缺口率 | 游泳辅助 |
| F9 | drr_tail_ratio dRR 尾部比率 | 伪迹预筛后 P95(\|dRR\|)/median(\|dRR\|)，RR 差分右尾厚重度 | 游泳/爬山上尾重、骑行/跑步平滑 |

附加量 `double_rr_rate`：RRᵢ ≈ 2×邻拍(±20%) 占比，用于信号门控与 dRR 伪迹预筛（不计入 9 特征打分）。

附加量 `climb` 会话内爬升幅度（后 1/3 段均值 − 前 1/3 段均值 bpm）：仅作诊断透传，**不计入打分**。曾尝试给跑步项加 `climb` 加权以捕捉「从低到高」爬升，但回归发现骑行上山同样有正爬升，会把标注为骑行的 0709-1 误判成跑步——`climb` 非跑步专属特征，故已回退加权。

### 6.2 信号质量门控（低置信兜底）
满足任一即强制 `low_confidence=True`、`dominant="混合"`、`poor_signal=True`，**不输出具体模式概率**（scores 置空、4 模式概率写 0）：
- `gap_ratio > 0.2`（真实中途掉线 / 严重缺口）
- `double_rr_rate > 0.05`（双 RR / 漏搏伪迹过多）

### 6.3 打分与概率
- 9 特征经各模式加权打分（登山重结构分+双峰、骑行重负荷+低间歇、跑步重负荷+漂移、游泳反负荷反平稳+高缺口），`softmax` 归一化为 5 模式概率。
- 低置信触发：`margin < 0.12`（最大两模式概率差过小）或 `p_max < 0.34`（最高概率过低）→ `low_confidence=True`、`dominant="混合"`。
- 置信度等级：高 / 中 / 低。报告徽章配色与 `MODE_COLORS` 一致（跑步红 / 骑行蓝 / 游泳绿 / 爬山橙 / 混合紫）。
- **间歇运动难分标记 `interval_ambiguous`**：当样本呈「高间歇 + 高尖峰 + 无漂移」形态（典型如高强度间歇跑/骑行，二者心率形态高度重叠）时置 True。此时即便概率 ≥0.6，报告徽章也从「高置信」**降级为「中置信(难分)」**，避免武断高置信判骑行。

### 6.4 报告卡片呈现规则
- **位置**：运动负荷分布下方（运动场景才有，睡眠不渲染）。
- **标题**：运动模式识别：{模式} {可信度徽章}（模式标签位于标题与徽章之间）。
- **描述句**：基于逐拍 RR 间期 / 瞬时心率的时域形态学特征（间歇度、尖峰率、平稳度、双峰性、平均负荷、HR 漂移等）做启发式估算。
- **图表**：概率条形图 `modeChart`（5 模式真实概率条）。
- **橙色 GPS 提示**：低置信 + 中置信显示（高置信不显示）。文案：「各模式概率接近或信号不足，区分度有限，结果仅供参考。真正判定需结合 GPS / 海拔 / 踏频 / 加速度计等传感器数据。」
- **蓝色「难分」提示**：`interval_ambiguous=True` 时额外显示。文案：「本段呈高强度间歇运动特征，跑步/骑行心率形态高度重叠，仅凭心率难以区分，建议融合踏频/GPS。」

## 七、心内科分析引擎说明
`CardioAnalyzer` 类内置于 `generate_report.py`：

### 信号质量评估
自动检测传感器伪差特征：运动中 RMSSD 异常偏高（>60ms）、高 SDNN 低三角指数分离（SDNN>80 且三角指数<12）、pNN50 与 pNN20 过于接近（比值>0.9）、精确 50%/100% RR 突变比例（倍半伪差模式）。输出可信度评级：高/中/低。

### HRV 指标解读
结合平均心率水平，对每项指标给出运动场景下的正常/异常判断（运动中 SDNN 通常 30-80ms、RMSSD 通常 <40ms、pNN50 通常 <3%、CVRR 通常 5-12%）。

### 异常事件分析
区分生理性 RR 波动（运动强度转换）与传感器伪差（精确倍半关系）；早搏临床风险评估（阈值 >1% 或 >100 次/小时）；异常事件时间集中性分析。

## 八、常见问题排查
1. 无 ch=2A37 数据：日志不含 XOSS 心率 BLE.RX 记录
2. 心率数值异常：脚本自动过滤 30~220bpm 外噪声
3. 固件版本未提取：日志中需包含 `fw=` 或 `firmware=` 字段
4. xlsx 打开乱码：确保使用 Excel 或 WPS 打开，非文本编辑器
5. **HTML 图表不显示（V2.4.0 已内联，通常不再发生）**：报告已内联 Chart.js，离线/沙箱预览应直接出图。若仍空白，先排查是否旧报告（含 `cdn.jsdelivr` 外链）；新报告应是 0 个 CDN 引用、含内联 `<script>window.Chart=...</script>`。空白多为 JS 语法错误导致整段中断——用 `node -e "new Function(html.match(/<script>([\s\S]*?)<\/script>/g)...) "`做语法校验，确认运动场景应有 7 个 `new Chart` 实例、睡眠 6 个。
6. **HTML 报告心内科分析为空**：检查 JSON 是否含完整 `hrv_metrics` 字段
7. **运动负荷评估文案与实际区间不符（V2.3.1 已修复）**：评估应完全基于 `exercise_segments` 各区间实际占比动态归纳
8. **运动模式 100% 混合 / 无四模式概率（Plan A 已修复）**：旧口径 `gap_ratio` 用「首末 ch=2A37 报文」作墙钟，会把运动后静默/无 RR 收尾算进缺口，虚高触发信号门控。V2.4.0 改为「有效 RR 报文首→末跨度」作墙钟（剔除前后静默）。若仍见异常高的缺口率导致全员混合，先核对 `compute_features` 的 `real_dur_sec` 是否取自 `valid_rows` 首末 `time` 的 `_parse_two_times` 结果。
9. **运动模式概率偏低/误判骑行为混合**：属启发式分类器固有局限（各模式 HR 区间重叠大）；低置信属诚实降级，最终判定须融合 GPS/踏频/海拔/加速度计。
10. **跑步 `sustained_high` 四门控（V2.4.3）**：单靠 drift 无法同时救高变异型跑步（drift≈0 但 var=13、high=60）和挡住高负荷骑行（drift≈0 且 var=9），因此 V2.4.3 在跑步得分里追加 `+ 1.5 * sustained_pct * var_running * drift_gate * tail_gate`。四个乘子任一为 0 即让 sustained_high 归零；均为"形态比率 / 占 HRmax 比例"，不引入任何 bpm 绝对硬值，跨 HRmax 人群普适。若真实跑步再被误判骑行，先查这四项：`high_pct` 落在 25~55、`variability` 落在 4~8、`drift_pct` 不低于 -0.10、`drr_tail_ratio` 不高于 30。

11. **真实跑步被判成骑行（间歇跑）**：高强度间歇跑与间歇骑行的心率形态高度重叠（高间歇+高尖峰+零漂移），纯心率无法区分。V2.4.1 起对此形态置 `interval_ambiguous=True`，徽章降级「中置信(难分)」并显示蓝色提示，不再武断高置信判骑行。切割日志的后半段会截断漂移信号（drift≈0），可拼回完整段以恢复漂移特征。切勿再靠加 `climb` 加权强判——会误伤上山骑行样本。
13. **极限档占比虚高（V2.4.4 已修复）**：旧口径直接以 `hr_p95` 作 HRmax 缩放，日志未跑到极限时 P95 只有 155~167，`极限=0.9×hr_p95` 门槛落在 140~150，稳态跑步/骑行心率轻松越过，导致极限档 40~65% 虚假爆表。V2.4.4 改为三级策略（override → max(190, p95) → 190 兜底），跑步/骑行日常样本极限档回到 <5% 的合理区间。如果确定本人 HRmax 低于 190（老年人/女性/低训练水平），显式传 `--hr-max` 或写入 `_user_meta.json`。

12. 百分比/数值显示精度溢出（V2.2.4 已修复）：数值显示统一 `:.2f` 格式化。
